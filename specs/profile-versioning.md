# Profile Versioning Specification

## Overview

Track every change to public profiles, private profiles, and working memory with full attribution: who made the change, via what mechanism, and when. Enables audit trails, change review, and rollback.

## Motivation

Now that both PIs and delegates can edit profiles (via web app or Slack DM), and agents autonomously rewrite working memory and private profiles, we need a clear record of who changed what and how. Today, profile changes are fire-and-forget — the markdown file on disk is overwritten with no history.

## Data Model

### ProfileRevision (new table)

| Field | Type | Notes |
|---|---|---|
| id | uuid | Primary key |
| agent_registry_id | FK → AgentRegistry | Which agent's profile was changed |
| profile_type | enum: public, private, memory | Which profile file |
| content | text | Full markdown snapshot after the change |
| changed_by_user_id | FK → User | Nullable. The human who initiated the change. Null for agent-initiated or system-initiated changes. |
| mechanism | enum: web, slack_dm, agent, pipeline, monthly_refresh | How the change was made (see below) |
| change_summary | text | Nullable. Brief description (e.g., "Added standing instruction: prioritize aging collaborations") |
| created_at | timestamp | When the revision was created |

**Indexes:**
- `(agent_registry_id, profile_type, created_at DESC)` — fast lookup of revision history for a given profile
- `(changed_by_user_id)` — find all changes by a specific user

**No foreign key to SimulationRun.** Working memory updates happen during simulation runs, but the `created_at` timestamp is sufficient to correlate. Keeping this table independent of SimulationRun avoids nullable FK complexity for non-simulation changes.

### Mechanism Enum

| Value | Description | changed_by_user_id |
|---|---|---|
| `web` | PI or delegate edited the profile in the web app | The user who saved the form |
| `slack_dm` | PI or delegate sent a Slack DM instruction; agent LLM-rewrote the private profile | The PI/delegate who sent the DM |
| `agent` | Agent autonomously updated its working memory (post-thread-closure, post-PI-DM, post-proposal-review) | null |
| `pipeline` | Profile ingestion pipeline generated/regenerated the profile (initial seed, manual refresh) | null |
| `monthly_refresh` | PI accepted a pending profile from the monthly refresh cron | The user who accepted |

### change_summary Guidelines

The `change_summary` field should be a short, human-readable description. It is **not** a diff — diffs are computed on the fly from consecutive snapshots.

| Mechanism | Example change_summary |
|---|---|
| web | "Edited research summary and added 2 techniques" (auto-generated from changed fields for public profiles, or null for private profile free-text edits) |
| slack_dm | "PI instruction: prioritize aging collaborations" (extracted from the PI's DM message) |
| agent | "Updated after thread closure: su ↔ wiseman in #drug-repurposing" |
| pipeline | "Initial profile generation from ORCID + PubMed" |
| monthly_refresh | "Accepted pending profile with 3 new publications" |

For `web` edits to public profiles, the summary can be auto-generated by comparing the previous revision's structured fields (techniques, disease_areas, etc.) against the new values. For free-text edits (private profiles, research summary only changes), `change_summary` can be null.

## Write Paths to Instrument

Every code path that writes a profile markdown file must also create a `ProfileRevision` row. The revision is created **after** the file is successfully written to disk, in the same database transaction as any other DB updates.

### Public Profile

| Code path | File | Mechanism | changed_by_user_id |
|---|---|---|---|
| PI/delegate edits via agent dashboard | `src/routers/agent_page.py` — `save_public_profile()` | web | session user |
| PI edits via profile page | `src/routers/profile.py` — `save_profile()` | web | session user |
| User saves during onboarding | `src/routers/onboarding.py` — `save_profile()` | web | session user |
| Profile pipeline (initial generation, manual refresh) | `src/services/profile_pipeline.py` — `run_profile_pipeline()` | pipeline | null |

### Private Profile

| Code path | File | Mechanism | changed_by_user_id |
|---|---|---|---|
| PI/delegate edits via agent dashboard | `src/routers/agent_page.py` — `save_private_profile()` | web | session user |
| User saves during onboarding | `src/routers/onboarding.py` — `save_private_profile()` | web | session user |
| PI sends standing instruction via Slack DM | `src/agent/pi_handler.py` — `_handle_standing_instruction()` | slack_dm | look up user_id from PI's Slack ID |
| Profile pipeline (private profile seed accepted) | `src/services/profile_pipeline.py` | pipeline | null |

### Working Memory

| Code path | File | Mechanism | changed_by_user_id |
|---|---|---|---|
| Agent updates after thread closure | `src/agent/simulation.py` — `_update_agent_memory()` | agent | null |
| Agent updates after PI DM | `src/agent/simulation.py` — `_update_agent_memory()` | agent | null |
| Agent updates after proposal review | `src/agent/simulation.py` — `_update_agent_memory()` | agent | null |

## Service Layer

### `src/services/profile_versioning.py`

A single service module with one primary function:

```python
async def create_revision(
    db: AsyncSession,
    *,
    agent_registry_id: uuid.UUID,
    profile_type: str,          # "public", "private", or "memory"
    content: str,               # full markdown content
    changed_by_user_id: uuid.UUID | None = None,
    mechanism: str,             # "web", "slack_dm", "agent", "pipeline", "monthly_refresh"
    change_summary: str | None = None,
) -> ProfileRevision:
    ...
```

And a helper for retrieving history:

```python
async def get_revision_history(
    db: AsyncSession,
    *,
    agent_registry_id: uuid.UUID,
    profile_type: str,
    limit: int = 50,
) -> list[ProfileRevision]:
    ...
```

Callers pass the full content (which they already have — they're about to write it to disk). The service just persists it.

## Diff Computation

Diffs are **not stored** — they are computed on the fly when displaying revision history. This keeps the table simple and avoids redundant storage.

For display in the admin dashboard or profile history page:
- Use Python `difflib.unified_diff()` server-side to generate a unified diff between consecutive revisions
- Render as a simple colored diff view (green for additions, red for removals)

## Retention

No automatic pruning for now. Profiles are small markdown files (typically 1-5 KB), and changes are infrequent (a few per day at most across all agents). At 12 agents, even a year of heavy use would produce a manageable number of rows.

Revisit if the table exceeds 100K rows.

Add a "History" link/tab to the public and private profile edit pages at `/agent/{agent_id}/profile`. Shows:

## Migration

### `alembic/versions/0009_profile_revisions.py`

- Create `profile_revision` table with all columns and indexes
- Create `profiletype` and `changemechanism` PostgreSQL enums

### Backfill

After migration, create one initial revision per existing profile file:
- Read each `profiles/public/*.md`, `profiles/private/*.md`, and `profiles/memory/*.md`
- Insert a revision with `mechanism='pipeline'`, `changed_by_user_id=null`, `change_summary='Initial backfill from existing file'`
- This ensures the diff view works from day one (there's always a "previous" revision to diff against)

The backfill can be a CLI command: `python -m src.cli backfill-profile-revisions`

## Not in Scope

- **Rollback UI:** Users cannot revert to a previous revision via the web app. If needed, an admin can manually copy content from a revision. Rollback can be added later.
- **Diffing structured fields:** For public profiles, we store the full markdown export, not individual field diffs. This is simpler and consistent across profile types.
- **Rate limiting revisions:** Every save creates a revision. No deduplication or debouncing.
