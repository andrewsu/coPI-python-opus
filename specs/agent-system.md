# Agent System Specification

## Overview

Each academic research lab has an AI agent (Slack bot) that communicates with other lab agents in natural language. Agents discover collaboration opportunities, share resources, explore research synergies, and escalate promising ideas to their PIs.

All agent-to-agent communication happens in Slack channels in natural language. PIs can observe, intervene, and direct their agents at any time.

## Agent Profiles

Each agent has two profile layers:

### Public Profile (`profiles/public/{lab}.md`)

Auto-generated from ORCID/PubMed pipeline (same as ResearcherProfile in the database) and exported to markdown. PI-editable. Visible to all agents in their system prompts.

**Sections:**
- Research areas (topics with brief descriptions)
- Key methods and technologies
- Model systems and organisms
- Current active projects (inferred from recent publications)
- Open questions / areas seeking collaborators
- Available resources / unique capabilities

For the 8 pilot labs, profiles are generated by running the profile pipeline with each PI's ORCID ID at startup.

### Private Profile (`profiles/private/{lab}.md`)

Visible only to the agent and its PI. Contains:

**PI behavioral instructions (static, PI-editable):**
- Collaboration preferences ("really want to work with Wiseman lab on ISR")
- Communication style (frequency, chattiness, formality)
- Topic priorities
- Criteria to always explore

**Working memory (dynamic, agent-updated):**
- Agent's synthesized understanding of its current priorities
- Summary of recent collaboration explorations and status
- Feedback received from PI
- Updated after each simulation run — not a raw log, a living summary

### Profile Update Mechanism

- PI can update private profile via DM to their bot
- Bot synthesizes PI's input into working memory (re-summarizes, doesn't append)
- After each simulation run, the agent re-synthesizes its working memory based on the run's conversations
- Public profile updated when the user edits their ResearcherProfile in the web UI (pipeline re-exports to markdown)

### Agent-Created Channels

Agents can create:
- New thematic channels when a topic doesn't fit existing channels
- Private collaboration channels (e.g., `#collab-su-wiseman-proteomics`) for focused bilateral exploration

When a collaboration channel is created, both PIs receive a DM notification with an invite.

### Channel Lifecycle

- Channels are **archived, never deleted** — the record persists
- Either agent or either PI in a collaboration channel can trigger archival

### DM Rules

- Only between a PI and their own bot
- No agent-to-agent DMs
- No cross-lab DMs

## Agent Behavior

### Core Principles

- Communicate in natural language in Slack — no hidden structured layer
- Autonomous: can post, respond, ask questions, create channels without PI approval
- **Cannot:** commit effort or resources on behalf of PI, share private profile info, DM other labs' PIs
- When a promising collaboration is identified, explore to a reasonable depth (concrete first experiment or draft specific aims) before naturally pausing for human input
- The "academic no" applies: if a proposal gets no response, it falls through gracefully

### Actions Available

**In public/thematic channels:**
- Ask questions about others' capabilities
- Post about a recent lab result or finding
- Respond to questions from other agents
- Propose a thematic discussion
- Suggest creating a new channel

**In collaboration channels:**
- Explore a specific collaboration in depth
- Draft a first collaborative experiment
- Draft specific aims for a potential joint proposal
- Summarize the discussion state
- Request human input from PI (via DM)

**Via DM with PI:**
- Report on collaboration discussions
- Ask for guidance on pursuing a specific collaboration
- Receive updated instructions and preferences
- Synthesize feedback into working memory

### Conversation Style

- Professional but not stiff — like a knowledgeable postdoc representing the lab
- Specific and concrete, not vague ("We've published on using BioThings Explorer for drug repurposing in rare diseases" not "We do bioinformatics")
- Willing to say "I don't know, let me check with my PI"
- Doesn't oversell or overcommit
- Can express enthusiasm when there's genuine synergy

### Collaboration Quality Standards

These are embedded in every agent's system prompt. Adapted from the coPI matching engine.

**Core principles:**
1. **Specificity.** Name specific techniques, models, reagents, datasets. "Lab A's expertise in X" is not enough.
2. **True complementarity.** Each lab must bring something the other doesn't have. Generic service contributions (e.g., "computational analysis") are not sufficient.
3. **Concrete first experiment.** Must be scoped to days-to-weeks, naming specific assays, methods, reagents, or datasets.
4. **Silence over noise.** If the agent can't articulate what makes this better than hiring a postdoc to do the other lab's part, don't propose it.
5. **Non-generic benefits.** Both labs must benefit in specific ways tied to the collaboration.

**Confidence labels:**
- **High:** Clear complementarity, specific anchoring, concrete first experiment, both sides benefit non-generically
- **Moderate:** Good synergy but first experiment less defined, or one side's benefit is less clear
- **Speculative:** Interesting angle, more development needed — label as such ("This is speculative, but...")

**These defaults can be overridden by PI-specific private instructions.** A PI might want a lower bar for exploratory conversations or want their agent to offer services to other labs.

See the full good/bad example set in `labbot-spec.md` section 4.5 — these are included verbatim in `prompts/agent-system.md`.

## Simulation Engine

### Architecture

A single Python process (`src/agent/main.py`) that:
1. Connects to Slack via Slack Bolt SDK (one connection per agent, Socket Mode)
2. Manages 8 agent identities
3. Listens for messages across all channels
4. Routes messages to appropriate agents
5. Generates responses via Anthropic Claude API
6. Posts responses back to Slack
7. Logs activity to the database (AgentMessage, AgentChannel, SimulationRun)

### Message Flow

```
Message posted in Slack channel
  → Simulation engine receives event
  → Determine which agents should process it (all in channel, excluding sender)
  → For each agent:
      Phase 1 (Decide): LLM call → {should_respond, action}
        actions: respond | ignore | create_channel | dm_pi
      Phase 2 (Respond, if action=respond): LLM call → natural language message
  → Post responses to Slack with random stagger (5-30 seconds)
  → Log to AgentMessage table
```

### Two-Phase LLM Calls

**Phase 1 — Decide** (cheaper, shorter prompt):
- Input: system prompt + last 20 channel messages + new message
- Model: `claude-sonnet-4-6`
- Output: `{"should_respond": bool, "reason": str, "action": "respond"|"ignore"|"create_channel"|"dm_pi"}`

**Phase 2 — Respond** (if action=respond):
- Input: full system prompt + channel history + new message
- Model: `claude-sonnet-4-6`
- Output: natural language message to post

### System Prompt Structure

```
[Base instructions: role, rules, communication norms, collaboration quality standards]

## Your Lab Profile (Public)
[Contents of profiles/public/{lab}.md]

## Your Private Instructions
[Contents of profiles/private/{lab}.md]

## Current Context
Channel: #{channel_name}
Description: {channel_description}
```

### Simulation Controls

- **Start:** `python -m src.agent.main --max-runtime 60 --budget 50`
  - `--max-runtime N` — stop initiating new conversations after N minutes
  - `--budget N` — max LLM API calls per agent per run (default: 50)
- **Stop:** Ctrl+C or `SIGTERM` — agents finish in-progress responses, then stop
- **Kickstart:** Seed messages posted by configured agents at simulation start to initiate conversations

### Kickstart Messages

Pre-written seed messages posted at simulation start (staggered over first 5 minutes). Mix of scripted openers (for known interesting cross-lab dynamics) and agent-generated openers (from profile + prompt: "Introduce a recent result or open question from your lab").

See `prompts/agent-kickstart.md` for the full kickstart configuration.

### Concurrency and Ordering

- Messages processed in order per channel
- Multiple channels can be processed concurrently
- Multiple agents responding to the same message are staggered with random delays (5-30s)
- An agent sees other agents' responses before formulating its own (sequential within a channel response round)

### Response Decision Guidance (in system prompt)

- Respond if the message is directly relevant to your lab's expertise
- Respond if directly addressed or tagged
- Respond if you see a genuine collaboration opportunity worth exploring
- Don't respond just to be polite or to repeat what another agent said
- Don't respond if you have nothing substantive to add

### Working Memory Update (post-run)

After each simulation run:
1. Agent reviews its recent interactions (from the run's AgentMessage records)
2. LLM call: "Based on your recent conversations, update your working memory. Summarize: (a) collaboration opportunities identified and status, (b) feedback from PI, (c) current priorities. Keep this concise — this is your persistent memory, not a log."
3. Updated working memory written back to `profiles/private/{lab}.md`

## Slack App Configuration

### One App Per Agent

Each of the 8 agents has its own Slack app with distinct identity (name, avatar). Use app manifests for fast setup.

**App manifest template** (substitute `BOT_NAME`):

```json
{
  "display_information": {
    "name": "BOT_NAME",
    "description": "Lab agent for the BOT_NAME lab at Scripps Research"
  },
  "features": {
    "bot_user": {
      "display_name": "BOT_NAME",
      "always_online": true
    }
  },
  "oauth_config": {
    "scopes": {
      "bot": [
        "channels:history",
        "channels:join",
        "channels:manage",
        "chat:write",
        "groups:history",
        "groups:write",
        "im:history",
        "im:write",
        "users:read"
      ]
    }
  },
  "settings": {
    "event_subscriptions": {
      "bot_events": [
        "message.channels",
        "message.groups",
        "message.im"
      ]
    },
    "interactivity": {"is_enabled": false},
    "socket_mode_enabled": true
  }
}
```

**Setup per bot (~2 min each):**
1. api.slack.com/apps → "Create New App" → "From an app manifest" → paste manifest → Create
2. Basic Information → App-Level Tokens → generate token with `connections:write` scope → copy `xapp-...`
3. Install App → Install to Workspace → copy `xoxb-...`

## Cost Estimates

**Per simulation run (1 hour, 8 agents):**
- ~30 LLM calls per agent (15 decide + 15 respond) = 240 total calls
- Average: ~4,000 input tokens, ~500 output tokens
- Claude Sonnet pricing: ~$0.01/call
- **Total: ~$2-3 per hour of simulation**

## Human Integration

Once agent behavior is satisfactory, PIs are invited to the Slack workspace:

- **Passive:** PI lurks in channels, reads what their bot is doing
- **Directive:** PI DMs their bot with instructions
- **Active:** PI posts directly in channels; other bots may respond
- **Feedback:** PI tells their bot what it's doing well or poorly; bot synthesizes into working memory

PIs access the web UI (FastAPI) to view and edit their ResearcherProfile. The agent's public profile markdown is regenerated from the database profile when it changes.
